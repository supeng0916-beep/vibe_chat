## 赛题名称：

VibeChat — AI 驱动的情绪社交

---

### 一、背景

很多线上社交产品以身份、兴趣、位置或关系链作为匹配依据，但人在真正想交流时，常常先需要被当前情绪理解。

这次赛题要求你完成一款基于情绪识别的匿名社交应用。用户输入一段文字后，系统通过 AI 分析其中的情绪色彩，再将情绪状态相近的用户自动匹配到同一段匿名对话中。它需要像一个轻量可用的社交产品，也需要展示你如何用 AI 把“情绪”转化成真实的匹配、对话和体验。

项目参考目标：使用 vibecoding 前后端在一天内开发完成，重点考验从产品定义、工程落地到线上部署的完整交付能力。

---

### 二、核心命题

> 做一个 AI 驱动的情绪社交 Web 应用，让用户基于当下情绪被理解、被匹配，并进入匿名对话。

本题的关键在于三件事：

- **识别情绪**：从用户文字中判断情绪类型、强度、倾向或复合情绪。

- **完成匹配**：根据情绪相似度，将用户自动匹配到合适的匿名聊天对象或聊天房间。

- **形成体验**：让用户感到这个产品围绕情绪展开，普通聊天室加 AI 标签无法覆盖本题要求。

---

### 三、必做要求

1. **情绪输入与识别**

    - 用户可以输入一段当前心情、状态或想说的话。

    - 系统需要调用 LLM API 对文字进行情绪分析。

    - 分析结果至少包含情绪标签和简短解释，建议包含情绪强度、正负向、关键词或情绪色彩。

2. **情绪匹配机制**

    - 系统需要根据情绪分析结果进行自动匹配。

    - 匹配逻辑需体现“相似情绪状态”这一核心规则。

    - 用户进入匹配后，应能看到匿名聊天对象或匿名房间。

    - 未匹配到用户的兜底机制。

3. **匿名对话能力**

    - 用户完成匹配后可以进行匿名文本对话。

    - 对话界面需具备基本聊天体验，包括消息发送、消息展示、时间顺序和当前会话状态。

    - 需要避免直接暴露真实身份信息，昵称、头像或身份标识可由系统生成。

4. **LLM API 适配**

    - 必须支持 **OpenAI 标准接口**。

    - 必须支持 **Anthropic 标准接口**。

    - 两种接口需要通过环境变量、配置项或后台设置切换。

    - README 中必须说明两种模式的配置方式，包括关键环境变量、模型名称和启动方式。

5. **前后端完整交付**

    - 技术栈：**FastAPI \+ Next\.js 前后端分离**。

    - 后端需提供情绪分析、匹配、会话和消息相关 API。

    - 前端需完成从情绪输入、结果展示、匹配等待到匿名聊天的完整流程。

6. **线上演示地址**

    - 必须部署到公网，并提供可访问的线上演示地址。

    - 评审时以线上演示版本为主要验收对象。

    - 如演示需要测试账号、临时密钥或特殊说明，需在提交材料中写清楚。

7. **可稳定演示**

    - 比赛提交时需要保证核心流程可连续跑通。

    - 如外部 LLM API 出现异常，产品应有清晰错误提示或降级处理。

    - 不能只提交静态页面或不可运行原型。

    - 针对聊天功能的演示，你可能需要 mock 不同用户进行聊天。

---

### 四、开放发挥

除必做要求外，不设标准答案。可以从以下方向发挥：

- 情绪可视化：用颜色、动效、图形或状态卡片呈现用户情绪。

- 匹配策略：支持相似情绪、互补情绪、同频房间、多人情绪聊天室等模式。

- 对话辅助：AI 生成开场白、温和提醒、情绪总结或对话破冰建议。

- 安全边界：识别极端负面情绪时给出求助提示，避免产品误导用户。

- 记忆与回顾：保存用户历史情绪轨迹、聊天摘要或情绪变化。

- 产品气质：让界面、文案和交互真正服务于“情绪社交”的主题。

评审会重点看：这个作品是否真的把 AI 情绪识别用于社交匹配，是否完整可用，是否有让人愿意继续体验的产品感。

### 八、比较严重的扣分项

- 无线上演示地址，或线上地址无法访问。

- 只支持 OpenAI 或只支持 Anthropic，未完成双标准适配。

- 情绪分析与匹配逻辑脱节，只展示情绪标签但未影响实际匹配

- 匿名聊天无法实际发送或接收消息
